① Group messaging — the core campaign

Groups were effectively broken (memo overflow, address drift, lost sends, forgeable control). This is the bulk of the work: invites that fit, messages that deliver and decrypt, and roster/key mutations that can't be forged.

Address-representation drift broke group messaging Critical Fix

#214 · #205 (root) · commit d5ca43b4b
WhatA single wallet emits several valid Unified-Address representations. Group roster/accept used exact-string matching, so a peer's "accept" never matched their invited rep — rosters never synced and messages never fanned out.
Why it happenedUA encodings differ between the receive screen and the KEX-sign path; #205 traced this as the root cause behind a whole family of address bugs.
FixPersisted self-address registry with hash-tolerant isSelfAddress(); peer-address alias registry so all reps of one peer collapse to one identity.

Test case TC-214

PrecondTwo funded wallets (Honor=admin, Seeker=member), v2.11.0 code 78.
Steps
  1. On Honor create a group, invite Seeker.
  2. On Seeker accept the invite.
  3. On Honor open Group Settings → members.
ExpectedSeeker shows as ACTIVE exactly once (no duplicate rows); a message sent from Honor is delivered to Seeker's live address.

GROUP_MSG decrypted with the wrong epoch key Critical Fix

#215 · commit a0a5ee0e9
WhatMessages were decrypted with the device's current group-key epoch, so any message sent before a key rotation (or from a lagging peer) showed "[Unable to decrypt]" forever.
FixDecrypt with the key for the message's own epoch (carried in the payload), with a bounded fallback window. Applied on both the live and history-load paths.

Test case TC-215

Steps
  1. In a group, exchange a few messages.
  2. Admin rotates the group key (kick or manual rotate).
  3. Scroll back to messages sent before the rotation; reopen the chat.
ExpectedPre-rotation messages still render decrypted (not "[Unable to decrypt]") after rotation and after reload.

Group send lost on single-note wallets Fix

#217 · commit 7fb155c14
WhatThe first group message right after invite/accept (which consumed the only note) failed with transient insufficient funds and was silently dropped.
FixBlock-aware retry: wait for change to confirm and retry; if delivered to ≥1 member clear the draft, else keep the draft and drop the optimistic bubble.

Test case TC-217

Steps
  1. Use a wallet with a single unspent note.
  2. Create group + invite (consumes the note).
  3. Immediately send a group message.
ExpectedMessage is NOT lost — it retries after change matures and delivers; draft preserved if delivery reaches no one.

Fan-out sent to stale invited address Fix + signed

#218 + #219 · commits c42a8c6a7, b7e6531a9, b539029a7
WhatGroup messages fanned out to the address a member was invited as, not their live receive address — and the address-adoption was unauthenticated (spoofable).
FixOn a verified GROUP_ACCEPT, adopt the accepter's declared receive address; sign the accept (#219) so adoption is proven by the key-holder. Alias registry holds a fixed-point invariant (#218 hardening).

Test case TC-218/219

Steps
  1. Member accepts from a wallet whose live address differs from the invited rep.
  2. Admin sends a group message.
  3. (Negative) Replay a forged unsigned accept with a different address.
ExpectedMessage reaches the member's live address; the forged/unsigned accept does NOT switch the fan-out target (safe degrade).

Signed GROUP_KICK / GROUP_KEY + admin UI Security Feature

#187 + #204
WhatKick and key-rotation control messages were unsigned → any on-chain party could evict a member or poison the group key.
FixPer-member signature with the admin's KEX key; receiver does authenticate → authorize (signer == admin) → epoch-replay-guard before acting. #204 added the admin UI trigger in Group Settings.

Test case TC-187

Steps
  1. As admin, Group Settings → Kick a member.
  2. (Negative) Inject an unsigned GROUP_KICK from a non-admin.
  3. (Negative) Replay an old signed kick (stale epoch).
ExpectedLegit kick removes the member & rotates the key; unsigned/forged kick is ignored; replayed stale-epoch kick is ignored.

Compact invite fits the 512-byte memo Critical Fix

#194 + #195
WhatGroup invites embedded the full roster + an ECIES blob and overflowed Zcash's 512-byte memo for any group size — invites silently failed (MemoTooLong) and groups never formed.
FixCompact invite carries only name + inviter + key_epoch + k2 (group key wrapped under the existing KEX session, ~432B); roster discovered lazily. #195 added a central pre-submit memo-size guard.

Test case TC-194

Steps
  1. Create a group and invite a KEX'd peer.
  2. Watch logcat / the send result.
ExpectedNo MemoTooLongException; the invite is submitted on-chain and the peer can join.

Pending↔mined reconciliation & retries Fix

#223 + #213 + #199 · commit 9dfc70a29
What(#223) Sent group messages stayed "pending" forever and other senders' messages could be hidden (seq-only dedup). (#213/#199) Accept/invite lost on transient insufficient funds.
FixReconcile by full (sender, seq, epoch) identity — keep all stored, drop only mined optimistic copies. Block-aware retry on accept/invite.

Test case TC-223

Steps
  1. Send a group message; wait for it to mine; reload the chat.
  2. Have another member send while yours is pending.
ExpectedYour message flips from pending→sent (no perpetual "sending"); the other member's message is NOT hidden.

② Tunnel / NOSTR / Calls

Tunnel mode (free, instant NOSTR after an on-chain handshake) and voice/video calls had a chain of bootstrap deadlocks. Reactions didn't propagate or persist.

Tunnel handshake bootstrap chain Fix

#200 · #206 · #208 · #209 · #201 · commit d247dd9f9
WhatA sequence of related deadlocks: ZBOOT never fired on fresh first-contact (#206); single-note retry window shorter than change maturation deadlocked the ZBOOT (#208); responder stayed in VAULT after bootstrap (#209); calls never unlocked because the NOSTR pubkey wasn't delivered (#200); superseded on-chain KEXs caused flapping (#201).
FixAlways (re-)send KEX at chat-open (idempotent); retry window > change maturation; routeIncomingBoot upgrades responder VAULT→TUNNEL; ignore superseded KEXs.

Test case TC-TUNNEL

Steps
  1. Honor: New Chat with Seeker in Tunnel mode.
  2. Wait for KEX → KEXACK → ZBOOT to complete.
  3. Send messages both directions; then place a call.
ExpectedBoth sides land in TUNNEL; messages free+instant over NOSTR; call buttons enabled and call connects.

Reactions propagate over NOSTR & persist Fix

#202 + #210 · commit ea2ebeaaa
WhatReactions didn't reach the recipient (target-id correlation mismatch) and were lost on reload.
FixCorrect target-id correlation; persist reactions so they survive reload and render on the recipient.

Test case TC-202

Steps
  1. React 👍 to a peer's message.
  2. Confirm it appears on the peer's device.
  3. Reload both chats.
ExpectedReaction shows on both sides and survives reload.

First-contact KEX exceeded 512-byte memo Critical Fix

#193
WhatThe first-contact KEX exceeded the memo limit (MemoTooLong) → "Unknown sender, can't reply".
FixTrimmed the KEX wire; it now carries an authenticated address the recipient recovers. Validated on a 2-device test.

Test case TC-193

StepsFirst-ever contact between two wallets; send an opening message.
ExpectedNo MemoTooLong; recipient can identify the sender and reply.

NOSTR receive-path dedup + rotation hot-swap Security

#188
WhatReplayed NOSTR events could be re-processed; inbox key rotation required a restart.
FixPersistent seen-event LRU dedup (chat + calls); inbox rotation hot-swap (no restart).

Test case TC-188

ExpectedDuplicate event ignored; rotation takes effect live.

#222 · commit 3ff57de5f
WhatWhile the seed phrase was visually blurred/hidden, the words were still exposed as plaintext in the accessibility tree (screen readers / a11y scrapers).

④ Performance — StrictMode main-thread I/O

Three disk/crypto operations were running on the UI thread (jank / ANR risk under StrictMode).

Move image-prep, file download/decrypt, and secure-wipe off the main thread Perf

What(#203) Image upload-prep, (#211) file download+decrypt+cache, and (#212) view-once secure-wipe all ran main-thread I/O — StrictMode violations.
FixAll three moved to background dispatchers.

Test case TC-PERF

PrecondStrictMode enabled (debug); logcat open.
Steps

  2. Download a received file.
  3. Open then dismiss a view-once item.
ExpectedNo StrictMode disk/crypto-on-main-thread violations; UI stays responsive.

⑦ /criticize adversarial hardening

A 3-round multi-agent self-audit caught a regression in the #215 fix and several latent issues — all fixed in the final build (code 78).

Bounded epoch loop, GK/GY delimiter guard, dead-code removal Critical Fix

/criticize R1–R3 · commit f560245dc
What(CRITICAL) The #215 epoch-aware decrypt iterated an unbounded epoch range with no exception guard — one bad stored key threw out the whole message and a high epoch could amplify work. Also: a canonical-string delimiter-injection surface on GK/GY, and two dead invite overloads.
FixBounded 64-epoch window + per-key runCatching on both decrypt paths; reject | in signer addresses; removed the dead overloads.

Test case TC-CRIT

Steps
  1. Receive a GROUP_MSG stamping a very high epoch number.
  2. Have one corrupt stored group key present.
ExpectedDecrypt attempts are bounded to ≤64; a corrupt key is skipped (not fatal); other messages still decrypt.
